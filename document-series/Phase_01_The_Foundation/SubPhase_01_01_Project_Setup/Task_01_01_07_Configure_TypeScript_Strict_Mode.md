# Task 01.01.07 — Configure TypeScript Strict Mode

## Metadata

| Field | Value |
|---|---|

| Estimated Complexity | Low |

## Objective

Update tsconfig.json to enable the full TypeScript strict mode flag set, configure the @/ path alias, and set moduleResolution to "bundler" for Next.js 15 compatibility, then confirm the compiler passes cleanly on the skeleton project.

## Instructions

### Step 1: Review the CLI-Generated tsconfig.json

Open tsconfig.json, which was generated by the Next.js CLI in Task 01.01.01. The CLI produces a reasonable baseline but enables only a subset of the strict-mode flags available in TypeScript. Before making any changes, read through the existing "compilerOptions" object carefully. Note which flags are already present (typically "strict": true and "noEmit": true are included by the CLI), and identify what is missing. This review prevents accidental duplication or contradiction when the additional flags are added.

### Step 2: Enable the Core Strict Flags

Inside "compilerOptions", confirm the "strict" master flag is set to true — this activates the baseline set of strict checks including strictNullChecks, strictFunctionTypes, strictBindCallApply, and strictPropertyInitialization. In addition to the master switch, explicitly list the following individual flags to make the project's intent clear to future contributors: set "noImplicitAny" to true to prevent TypeScript from silently inferring the any type in ambiguous situations, set "strictNullChecks" to true to ensure null and undefined cannot be assigned to non-nullable types without explicit handling, and set "noUncheckedIndexedAccess" to true to mark every array index access and string-keyed object lookup as returning T or undefined rather than just T. This last flag catches a large class of potential runtime errors where code assumes an array element or map value exists without a bounds check.

### Step 3: Add Correctness Flags Beyond the Strict Set

Add the following flags that are not included in the "strict" master switch but are strongly recommended for a production TypeScript project. Set "noImplicitReturns" to true to require that every code path in a function explicitly returns a value of the declared return type, preventing accidentally void functions. Set "noFallthroughCasesInSwitch" to true to require explicit break or return statements in every switch case, preventing silent fall-through bugs. Set "exactOptionalPropertyTypes" to true to prevent optional properties from being explicitly assigned the value undefined unless undefined is part of their declared type union — this enforces the distinction between a missing property and an explicitly undefined one. Set "forceConsistentCasingInFileNames" to true to prevent cross-platform bugs where a file imported as MyComponent.tsx on a case-insensitive macOS filesystem resolves differently on a case-sensitive Linux CI server.

### Step 4: Set moduleResolution to "bundler"

Locate the "moduleResolution" entry in compilerOptions. The Next.js CLI may have set this to "node", "node16", or "Bundler". Set it to "bundler" (with the correct capitalisation as required by TypeScript). This resolution mode is designed for applications built with a module bundler such as webpack or Turbopack, and it correctly supports the package.json exports field, TypeScript's own path aliases, and the ESM-style bare specifier imports that Next.js and its ecosystem packages use. Using the wrong moduleResolution can cause TypeScript to reject valid imports or fail to resolve type declarations from modern packages.

### Step 5: Verify the @/ Path Alias Configuration

Confirm that the "paths" object inside "compilerOptions" contains an entry for "@/*" mapped to the array value ["./src/*"]. The Next.js CLI typically configures this during project creation when the import alias question is answered with @/*, but manually verify it is correct. Also confirm that "baseUrl" is set to "." (a single dot representing the project root directory). The baseUrl is required for the paths configuration to resolve correctly — without it, TypeScript cannot determine the base directory from which path alias lookups begin.

### Step 6: Verify the include and exclude Arrays

Confirm that the top-level "include" array in tsconfig.json covers the file patterns "next-env.d.ts", "**/*.ts", and "**/*.tsx". The next-env.d.ts file is generated by Next.js and provides type augmentations for the framework. If "prisma/**/*" is not already in the include array, add it so that the seed script created in Task 01.01.11 is type-checked as part of the project. Confirm the "exclude" array includes "node_modules" to prevent the TypeScript compiler from attempting to type-check third-party package source files.

### Step 7: Run the Type Check and Resolve Any Errors

Run "pnpm tsc --noEmit" in the terminal. This runs the TypeScript type checker in its check-only mode without emitting any compiled JavaScript output. With the skeleton project containing minimal code, this should complete with zero errors. If errors appear — most likely from ShadCN component files added in Task 01.01.04 — investigate each error and apply the appropriate type annotation. Common issues at this stage include the noUncheckedIndexedAccess flag causing TypeScript to require undefined checks on array element accesses that were previously assumed to always yield a value.

### Step 8: Verify Path Alias Resolution in VS Code

Open any file in src/ and type an import statement using the @/ prefix, for example importing something from "@/lib/utils". Confirm that VS Code's TypeScript language service resolves the path without showing a red underline or a "Cannot find module" error in the Problems panel. The IntelliSense autocomplete should also show completions for @/components/, @/lib/, @/hooks/, @/stores/, and @/types/. Remove the test import once verified.

## Expected Output

- tsconfig.json has strict, noImplicitAny, strictNullChecks, noUncheckedIndexedAccess, noImplicitReturns, noFallthroughCasesInSwitch, exactOptionalPropertyTypes, and forceConsistentCasingInFileNames all set to true
- moduleResolution is set to "bundler"
- The @/* path alias is configured to resolve to ./src/* with baseUrl set to "."
- "pnpm tsc --noEmit" completes with zero errors on the skeleton project

## Validation

- [ ] "pnpm tsc --noEmit" exits with zero errors
- [ ] VS Code IntelliSense resolves @/components, @/lib, @/hooks, @/stores, and @/types without errors
- [ ] tsconfig.json contains all eight strict flags listed in Steps 2 and 3
- [ ] moduleResolution is set to "bundler" in compilerOptions
- [ ] prisma/**/* is in the include array

## Notes

The "noUncheckedIndexedAccess" flag is the most impactful of the added flags and will cause the most friction as the codebase grows. When this flag is active, every array element access returns T | undefined rather than T, which means any code that accesses array[0] or object[key] must handle the undefined case explicitly. This is the correct, safe behaviour for production code and prevents a wide category of runtime errors. All developers working on VelvetPOS should understand this flag's implications before writing TypeScript that performs array or map lookups.
