# Task 01.01.05 — Configure ESLint and Prettier

## Metadata

| Field | Value |

## Objective

Configure ESLint with Next.js and TypeScript strict rule sets, Prettier with VelvetPOS formatting standards, and Husky with lint-staged pre-commit hooks that block any commit where ESLint or Prettier checks fail.

## Instructions

### Step 1: Install ESLint Plugins and Dependencies

The Next.js CLI scaffold already installed "eslint" and "eslint-config-next" during Task 01.01.01. Now extend that baseline by installing the TypeScript ESLint toolchain: run "pnpm add -D @typescript-eslint/parser @typescript-eslint/eslint-plugin". These two packages provide TypeScript-aware linting rules that analyse type information beyond what Next.js's default ESLint configuration covers. Also run "pnpm add -D eslint-plugin-import" to add import ordering and grouping enforcement, which keeps the codebase consistent in how dependencies are imported across all files.

### Step 2: Configure the ESLint Flat Config

Open eslint.config.mjs. The Next.js CLI may have placed a minimal configuration here. Replace the entire file content with a properly structured flat configuration array. The array should include: the Next.js core web vitals configuration imported from "eslint-config-next/core-web-vitals", a TypeScript ESLint configuration object applying to all ".ts" and ".tsx" files that includes the "@typescript-eslint/recommended-type-checked" rule preset with the languageOptions parserOptions specifying project set to "./tsconfig.json" to enable type-aware linting, and a custom rules override object. In the rules override, set "@typescript-eslint/no-explicit-any" to "error" so any use of TypeScript's any type fails the lint check as an error rather than a warning. Set "@typescript-eslint/no-unused-vars" to "error" with options to check all variables and caughtErrors. Set "no-console" to "warn" so that console.log or console.error statements are flagged and must be addressed before production. Set "react/self-closing-comp" to "error" to enforce self-closing JSX tags on components with no children. Configure "import/order" to "warn" with groups ordering imports as builtin, external, internal, parent, sibling, and then index, with blank lines between each group.

### Step 3: Install and Configure Prettier

Run "pnpm add -D prettier prettier-plugin-tailwindcss" to install Prettier and the official Tailwind CSS class sorting plugin. Create a file named "prettier.config.mjs" in the project root (use the .mjs extension for ES module compatibility). The file should export a default configuration object containing the following settings: printWidth set to 100 (lines over 100 characters will be wrapped), tabWidth set to 2 (two-space indentation throughout all files), singleQuote set to true (single quotes are preferred over double quotes in all JavaScript and TypeScript expressions), trailingComma set to "all" (trailing commas are added to every multi-line construct including function parameter lists and destructuring patterns), semi set to true (semicolons are always included), and plugins set to an array containing the string "prettier-plugin-tailwindcss". Add a tailwindConfig key pointing to "./tailwind.config.ts" so the Tailwind plugin can read the VelvetPOS configuration and sort classes in the project's canonical token order.

### Step 4: Add Scripts to package.json

Open package.json and add the following entries to the "scripts" object. Add a "lint" script that runs "eslint src/ --max-warnings=0" — the --max-warnings flag causes ESLint to exit with a non-zero code if any warnings are present, effectively treating warnings as errors in CI contexts. Add a "lint:fix" script running "eslint src/ --fix" for auto-correctable issues. Add a "format" script running "prettier --write ." to format all files in the project. Add a "format:check" script running "prettier --check ." to verify formatting without making changes, useful in CI pipelines.

### Step 5: Install and Initialise Husky

Run "pnpm add -D husky lint-staged" to install both tools. Then run "pnpm exec husky init" to initialise Husky in the repository. This creates a .husky/ directory at the project root and automatically adds a "prepare" script to package.json that runs "husky" on every invocation of "pnpm install". The prepare hook ensures every developer who clones the repository and runs "pnpm install" gets Husky hooks activated automatically without any manual configuration step.

### Step 6: Configure the Pre-Commit Hook

Open the .husky/pre-commit file generated by Husky init. Replace its default placeholder content with a single line running "pnpm exec lint-staged". This delegates all pre-commit processing to lint-staged, which is configured in the next step. Keeping the pre-commit hook minimal and delegating to lint-staged is the recommended practice because it allows lint-staged config to be version-controlled in package.json and easily updated without touching the hook file.

### Step 7: Configure lint-staged in package.json

Open package.json and add a top-level "lint-staged" key. Inside it, configure rules for two file type patterns. Files matching "**/*.{ts,tsx}" should be processed by the two commands "eslint --fix" followed by "prettier --write" — in that order, because ESLint's auto-fix may change code that Prettier then needs to format. Files matching "**/*.{json,md,css}" should be processed only by "prettier --write". This targeted approach runs linting only on staged files rather than the entire codebase, keeping pre-commit execution time fast even as the project grows.

### Step 8: Verify the Full Toolchain

Run "pnpm lint" and confirm it exits with zero warnings and zero errors on the current skeleton codebase. Run "pnpm format:check" and confirm Prettier reports no formatting violations. Stage a test file with intentional style violations (such as double quotes or a missing trailing comma) and attempt a git commit — verify the pre-commit hook triggers, lint-staged runs, and the violations are either auto-fixed or the commit is blocked with a descriptive error message.

## Expected Output

- eslint.config.mjs enforces TypeScript strict rules, no-console warnings, no-explicit-any errors, and import ordering
- prettier.config.mjs enforces 2-space indent, single quotes, trailing commas, 100-character line limit, and Tailwind class sorting
- .husky/pre-commit is present, executable, and delegates to lint-staged
- package.json contains lint-staged configuration and four new scripts: lint, lint:fix, format, format:check
- "pnpm lint" and "pnpm format:check" both exit cleanly on the project skeleton

## Validation

- [ ] "pnpm lint" exits with zero warnings and zero errors
- [ ] "pnpm format:check" exits with no formatting violations
- [ ] A staged file with intentional formatting errors is caught by the pre-commit hook
- [ ] "pnpm tsc --noEmit" still passes after ESLint configuration is updated

## Notes

The "prepare" script that Husky adds to package.json is the mechanism that installs Git hooks automatically for all contributors. If a contributor's environment runs pnpm install with the --ignore-scripts flag (common in some CI environments), the prepare script will not run and hooks will not be activated. For CI pipelines, linting and formatting should be run explicitly as separate pipeline steps rather than relying on pre-commit hooks, which are a developer convenience rather than a CI enforcement mechanism.
