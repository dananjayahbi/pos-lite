# Task 01.03.01 — Create Tenant And Subscription Models

## Metadata

- **Sub-Phase:** 01.03 — SaaS Infrastructure & Tenant Management
- **Phase:** 01 — The Foundation
- **Estimated Complexity:** High
- **Dependencies:** SubPhase_01_01_Project_Setup complete, SubPhase_01_02_Auth_And_RBAC complete

## Objective

Add the Tenant, Plan, Subscription, and Invoice models along with their corresponding status enums to prisma/schema.prisma, update the User model to carry the formal Tenant relation, and execute a clean database migration so all new tables and constraints are provisioned and ready for the service layer to consume.

## Instructions

### Step 1: Open the Prisma Schema File

Navigate to prisma/schema.prisma in the project root. This file already contains the datasource and generator configuration blocks as well as the User, Role enum, and AuditLog model definitions created during SubPhases 01.01 and 01.02. All additions in this task are appended below the existing content. Nothing in the file's current content should be removed or restructured.

### Step 2: Declare the Three Status Enums

Before writing any new model block, declare three new Prisma enum types. The first is TenantStatus, which has four members: ACTIVE, GRACE_PERIOD, SUSPENDED, and CANCELLED. The second is SubscriptionStatus, which has four members: ACTIVE, PAST_DUE, CANCELLED, and TRIALING. The third is InvoiceStatus, which has three members: PAID, UNPAID, and OVERDUE. Each enum is a named block in the Prisma schema language containing its members listed one per line. These enums will be referenced as field types in the models that follow.

### Step 3: Define the Plan Model

Add the Plan model to the schema. The model must declare the following fields in order: id as a String type marked as the primary key with a default value generated by the cuid function; name as a non-nullable String with a unique constraint so no two plans share identical names; description as a non-nullable String used for the marketing copy displayed in the provisioning wizard; priceMonthly as a Decimal type with precision 12 and scale 2 representing the monthly cost in Sri Lankan Rupees; features as a Json type that stores an ordered array of feature description strings; isActive as a Boolean with a default value of true, allowing plans to be deactivated without deletion; sortOrder as an Int with a default value of zero, controlling display order in the wizard; and subscriptions as a list relation field referencing the Subscription model, marking this as the one side of the one-to-many relationship.

### Step 4: Define the Tenant Model

Add the Tenant model below the Plan model. The fields must be: id as the cuid String primary key; name as a non-nullable String containing the store's display name; slug as a non-nullable String with a unique constraint — this value becomes the subdomain identifier and must contain only URL-safe characters; logoUrl as an optional nullable String for the store's logo image URL; status as the TenantStatus enum with a default value of ACTIVE; graceEndsAt as a nullable DateTime marking when the grace period expires; customDomain as a nullable String for tenants that have registered a vanity domain; settings as a Json field storing the store's configuration object per the documented schema; createdAt as a DateTime with a default of now(); updatedAt as a DateTime using the Prisma updatedAt attribute for automatic timestamp management; and deletedAt as a nullable DateTime to support soft deletes without physical row removal. The relation fields are users as a list of User records, subscriptions as a list of Subscription records, and invoices as a list of Invoice records.

### Step 5: Define the Subscription Model

Add the Subscription model below the Tenant model. The fields are: id as the cuid String primary key; tenantId as a non-nullable String acting as the foreign key to the Tenant model; tenant as the relation field back to Tenant using the fields and references attributes pointing to tenantId and the Tenant id respectively; planId as a non-nullable String foreign key to the Plan model; plan as the relation field to Plan via planId; status as the SubscriptionStatus enum type; currentPeriodStart as a non-nullable DateTime marking the start of the active billing period; currentPeriodEnd as a non-nullable DateTime marking the end; payhereSubId as a nullable String used to store the external subscription identifier once PayHere integration is completed in Phase 5; nextBillingDate as a non-nullable DateTime for the scheduler to use when triggering renewal reminders; cancelledAt as a nullable DateTime populated when the subscription is cancelled; createdAt as a DateTime defaulting to now(); and updatedAt as a self-updating DateTime. The invoices field is a list relation to the Invoice model.

### Step 6: Define the Invoice Model

Add the Invoice model below the Subscription model. The fields are: id as the cuid String primary key; tenantId as a non-nullable String foreign key; tenant as the relation field to Tenant via tenantId; subscriptionId as a non-nullable String foreign key; subscription as the relation field to Subscription via subscriptionId; invoiceNumber as a non-nullable String with a unique constraint so each invoice's reference code is globally unique; amount as a Decimal with precision 12 and scale 2; status as the InvoiceStatus enum; billingDate as a non-nullable DateTime recording when the invoice was raised; paidAt as a nullable DateTime populated when payment is confirmed; pdfUrl as a nullable String for the future invoice PDF storage link; and createdAt as a DateTime defaulting to now().

### Step 7: Update the User Model With the Tenant Relation

Locate the User model earlier in the schema file. It already contains a tenantId field declared as a nullable String, which was added as a structural placeholder during SubPhase 01.01. Add the formal Prisma relation immediately after this field: a tenant field of the optional Tenant type decorated with the relation attribute specifying fields as [tenantId] and references as [id]. This upgrade from a bare foreign key column to a declared Prisma relation activates all relational query features in the generated client, including nested reads and writes.

### Step 8: Run the Database Migration

From the project root terminal, execute pnpm prisma migrate dev --name add_tenant_models. Prisma will compare the current database schema against the updated schema.prisma file, compute the diff, and write a new SQL migration file into prisma/migrations/ with a timestamped directory name. It then applies the migration to the development PostgreSQL database. When the command completes, open the generated SQL file and confirm it contains CREATE TABLE statements for plan, tenant, subscription, and invoice tables, an ALTER TABLE statement adding a foreign key from the users table to the tenant table, and the corresponding ENUM type definitions.

### Step 9: Regenerate the Prisma Client

Run pnpm prisma generate to rebuild the TypeScript client definitions. This step is sometimes performed automatically by the migrate dev command but should be run explicitly to guarantee the types are fresh. After generation, import the TenantStatus enum from the Prisma client package in a scratch editor and confirm TypeScript resolves it without errors before proceeding.

### Step 10: Verify Using Prisma Studio

Launch Prisma Studio by running pnpm prisma studio. In the browser tab that opens, confirm that six tables are visible: User, AuditLog, Plan, Tenant, Subscription, and Invoice. Click each new table and verify the column names match the schema definitions exactly. Close Prisma Studio once the check is complete.

## Expected Output

- A timestamped directory under prisma/migrations/ holds the generated SQL for all new tables and constraints
- pnpm tsc --noEmit produces no TypeScript errors
- Prisma Studio shows all six tables with correct columns

## Notes

The slug field on Tenant carries only a database-level uniqueness constraint. The format validation — ensuring slugs contain only lowercase letters, digits, and hyphens with no leading or trailing hyphens — must be enforced at the application layer inside the provisioning wizard's Zod validation schema. The settings Json column is stored by PostgreSQL as JSONB, which supports indexing on specific JSON keys in future phases. Document the canonical settings key names in the SubPhase Overview's Technical Context section so all future developers use consistent key names when reading or writing this field.
