## Objective

Add the Category and Brand Prisma models to prisma/schema.prisma, apply a named database migration, and verify both tables are visible and correctly structured in Prisma Studio.

---

## Instructions

### Step 1: Review the Existing Schema

Open prisma/schema.prisma and read through the models already defined from Phase 01. Locate the section where new Phase 02 models will be appended. Confirm that the datasource block specifies provider as postgresql and that the generator block targets prisma-client-js. Take note of how existing models use cuid() as the default for id fields and how DateTime fields use @default(now()) — follow the same conventions for consistency.

### Step 2: Define the Category Model

Add the Category model after the existing Phase 01 models. The model requires the following fields in order:

- id: a String field serving as the primary key, defaulting to a CUID generated by Prisma
- tenantId: a String that acts as a foreign key referencing the Tenant model — this is the primary multi-tenant isolation field
- name: a non-optional String representing the display name of the category
- description: an optional String for additional detail about the category
- parentId: a nullable String that is a self-referential foreign key pointing back to the Category model's own id — this enables nested category trees of up to two levels for the clothing store's needs
- sortOrder: an Int defaulting to 0, used to control display order in category lists and navigation menus
- createdAt: a DateTime defaulting to now()
- updatedAt: a DateTime that uses the @updatedAt Prisma attribute so it is automatically refreshed on every record update
- deletedAt: a nullable DateTime to support soft deletion — this field being non-null means the category is considered logically deleted

Add two relation fields: a parent field of type Category that is optional (using the @relation with the parentId field), and a children field that is an array of Category, representing the subcategories underneath this category. Additionally add a products relation field as an array of Product pointing to the categoryId field on the Product model — this will be resolved once the Product model is added in Task 02.01.02.

Attach the following indexes and unique constraints to the Category model:

A composite unique index on the combination of tenantId and name ensures no two active categories within the same tenant share the same name. This constraint is enforced at the Prisma schema level using the @@unique directive with [tenantId, name]. Note that this unique constraint does not account for soft-deleted records — application logic in the service layer must filter out deleted categories before validating uniqueness on name.

Add a standard index on tenantId alone using the @@index directive to support the most common query pattern of "fetch all categories for this tenant".

### Step 3: Define the Brand Model

Add the Brand model following the Category model. Its fields are:

- id: String, primary key, CUID default
- tenantId: String, multi-tenant isolation foreign key referencing Tenant
- name: non-optional String, the brand display name
- description: optional String for brand bio or notes
- logoUrl: optional String storing the public URL of the brand logo image (uploaded via the file storage abstraction in Task 02.01.11)
- createdAt: DateTime, default now()
- updatedAt: DateTime, @updatedAt
- deletedAt: nullable DateTime for soft deletion

A products relation field as an array of Product will be added once the Product model is defined.

Apply a composite unique constraint on [tenantId, name] to prevent duplicate brand names within a tenant. Apply an index on tenantId for efficient tenant-scoped queries.

### Step 4: Run the Migration

With both models fully defined in prisma/schema.prisma, run the Prisma migration command: pnpm prisma migrate dev --name add_category_and_brand_models. This will generate a new timestamped migration file under prisma/migrations/, compute the SQL diff against the current schema, and apply it to the development PostgreSQL database. Confirm that the terminal output shows the migration was applied successfully with no warnings.

### Step 5: Verify the Migration in Prisma Studio

Run pnpm prisma studio to open the Prisma Studio browser interface. Navigate to the Category table and confirm it is listed with all expected columns: id, tenantId, name, description, parentId, sortOrder, createdAt, updatedAt, and deletedAt. Do the same for the Brand table. Verify the columns match the schema definition. Close Prisma Studio once verification is complete.

### Step 6: Regenerate the Prisma Client

After migration, run pnpm prisma generate to regenerate the Prisma Client. This ensures that TypeScript types for Category and Brand are available in the codebase. Confirm that the generation completes without errors. The updated client will be used starting in Task 02.01.06 when the service layer is built.

## Notes

The soft delete pattern in VelvetPOS uses a nullable deletedAt DateTime. A non-null deletedAt signals that the record is logically deleted. This means all queries in the service layer must include a where clause filtering for deletedAt being null (meaning "not deleted"). Prisma does not support automatic global soft-delete filtering natively — each query must explicitly filter. This is enforced by code review convention.

The self-referential parentId on Category is designed for a shallow two-level hierarchy. The system does not need to support trees of arbitrary depth, so there is no recursive CTE or closure table required. Categories that serve as parents should have a null parentId themselves. Attempting to set a category as its own parent must be prevented in the service layer validation.

Composite unique constraints in Prisma using @@unique do not natively support partial uniqueness (i.e., excluding rows where deletedAt is non-null). This is an accepted trade-off: re-using a deleted category's name for a new category requires unsetting deletedAt on the original record rather than creating a new one, or the service layer must check for soft-deleted conflicts and return a clear error message suggesting the existing archived category be restored instead.
