## Objective

Implement a hardware barcode scanner integration as a custom React hook that captures the high-speed keystroke sequences generated by USB HID barcode scanners, resolves the scanned barcode against the product catalog, and adds the matching variant to the cart with appropriate user feedback.

## Instructions

### Step 1: Understand How Hardware Barcode Scanners Work

USB HID (Human Interface Device) barcode scanners present themselves to the operating system as a keyboard. When a barcode is scanned, the scanner emits the barcode string as a sequence of keydown and keyup events at a speed far beyond human typing capability — typically 50 to 200 characters per second — followed by an Enter keypress. From the browser's perspective, these events are indistinguishable from rapid keyboard input, which is why no special device drivers or browser APIs are required.

The distinguishing characteristic between scanner input and human typing is speed. A scanner completes the entire barcode sequence (typically 8 to 15 characters) within 50 to 100 milliseconds total. A fast human typist may produce a keystroke every 70 to 120 milliseconds per key. The barcode hook exploits this timing difference: if two consecutive keydown events occur within 50 milliseconds of each other, they are assumed to originate from a scanner rather than a human. This threshold is stored as a configurable constant.

### Step 2: Design the useBarcodeScanner Hook

Create src/hooks/useBarcodeScanner.ts. The hook accepts a configuration object containing tenantId (string), onAdd (a callback invoked when a variant is successfully resolved and added to the cart, receiving the variant id and product name for notification), and enabled (boolean defaulting to true — the hook can be temporarily disabled, for example when a modal is open).

The hook maintains three pieces of internal state managed via useRef (not useState, to avoid re-renders on every keystroke): a buffer string accumulating characters from the current scan sequence, a lastKeyTime timestamp storing when the most recent keydown event occurred, and a flushTimeoutRef storing the id of the pending flush timeout.

### Step 3: Implement the Event Listener

Use a useEffect to attach the event listener on mount and remove it on unmount. Attach the listener to the window object using addEventListener for the "keydown" event. In the cleanup function of the useEffect, call window.removeEventListener with the same handler reference. This cleanup is essential to prevent memory leaks and duplicate listener registration caused by React's StrictMode double-invocation in development, or by the component unmounting and remounting during navigation.

Inside the keydown handler, perform the following checks before processing the event. First, check document.activeElement: if the currently focused element is an HTMLInputElement, HTMLTextAreaElement, or an element with contentEditable set to true, return immediately without processing the keystroke. This prevents the barcode listener from capturing characters while the cashier is typing in the search input, the discount control input, or any other text field. This conflict-prevention check is critical — without it, every character the cashier types in a text field would be buffered and potentially misinterpreted as a barcode.

Second, check whether the enabled prop is true. If the hook has been disabled by the parent, return without processing.

### Step 4: Buffer Accumulation and Timing Logic

When a non-special key is pressed (a printable character — check that key.length is 1, which filters out "Shift", "Control", "Enter", and similar special key names), compute the elapsed time since the last keydown event in milliseconds. If the elapsed time is less than or equal to 50 milliseconds, the keystroke is classified as part of an ongoing scanner sequence. Append the key to the buffer string and update lastKeyTime to the current timestamp. Cancel any pending flush timeout (using clearTimeout on flushTimeoutRef.current) and set a new flush timeout for 100 milliseconds to handle the end of a scan sequence that is not terminated by Enter.

If the elapsed time exceeds 50 milliseconds, the keystroke is more likely a human keypress. If the buffer is currently empty (no scan in progress), this is just a regular human keystroke — do nothing (allow the keydown event to propagate normally, which is important for shortcut keys and other terminal-level keyboard interactions). If the buffer is non-empty, the slow keystroke may be erroneous; clear the buffer and start fresh.

When the "Enter" key is pressed, regardless of the elapsed time since the last character, treat the buffered string as a completed scan if its length is 6 or more characters. Call the flush function with the current buffer contents and clear the buffer.

### Step 5: Implement the Flush and Resolve Logic

The flush function receives the buffered barcode string. It first trims any whitespace and validates the string length meets a minimum of 6 characters — discard shorter strings as noise. Call getVariantByBarcode from the product service using TanStack Query's queryClient.fetchQuery. Use a dedicated query key such as ["variant-by-barcode", barcode, tenantId]. The fetchQuery call first checks the local query cache for a matching entry; only if no cached entry is found does it make an API request to GET /api/products/by-barcode?sku=[barcode]&tenantId=[tenantId]. This cache-first approach means repeatedly scanning the same popular product triggers zero additional network requests.

Handle the response from fetchQuery. If the variant is found and its stock quantity is greater than zero (or the tenant's allowNegativeStock setting is true), call the useCartStore addItem mutator with the resolved variant's information — populating all snapshot fields from the returned variant and product data. Then call the onAdd callback with the variant id and product name so the parent component can show a flash notification. If the variant is found but stock is zero, trigger a warning notification directly from within the hook using a toast library (such as sonner or the ShadCN toast utility imported into the hook) with the message "Out of stock: [Product Name] [Variant]" in the warning colour. Do not add the variant to the cart. If no variant is found matching the barcode, trigger a danger toast reading "Unknown barcode: [scannedCode]" providing the cashier with the exact scanned string so they can verify the barcode label is not damaged.

### Step 6: Integrate the Hook into the POS Page

In the POS terminal page (src/app/dashboard/[tenantSlug]/pos/page.tsx), call useBarcodeScanner at the top level of the component, passing the tenantId and an onAdd callback that triggers a brief "Added" flash notification below the search bar. Disable the hook when the VariantSelectionModal or CartManagerPINModal is open by reading their open state and passing it as the enabled prop value set to false. This prevents the barcode scanner from attempting to add items while a modal is blocking the terminal's primary interaction layer.

### Step 7: Build the Flash Notification Area

In the POS left panel, directly below the search input and above the CategoryTabs component, render a fixed-height (approximately 28px) notification strip. This strip is hidden when no notification is active. When a successful scan occurs, the strip slides down with a green (#2D6A4F) background showing the text "Added: [Product Name] [Size/Colour]" in white Inter 13px. The notification auto-dismisses after 2.5 seconds. When an out-of-stock or unknown barcode occurs, the strip uses a warning or danger background respectively with the corresponding message. Only one notification occupies the strip at a time — a new scan replaces any in-progress notification.

## Validation

- Simulating a barcode scan by rapidly typing a known SKU string followed by Enter (using the browser console's dispatchEvent or a test automation script) adds the correct variant to the cart and shows a green flash notification
- Typing the same SKU slowly (more than 50ms between each character) while no text field is focused does not trigger the barcode handler — each keystroke is processed independently as a normal keypress
- Typing in the search input field while a scanner scan is occurring does not cause the scanned characters to accumulate in the barcode buffer — the active element check correctly suppresses the handler
- Scanning a barcode for a product with zero stock shows the warning flash and does not add to cart
- Scanning an unrecognised barcode string shows the danger flash with the scanned string displayed
- Unmounting and remounting the POS page (simulated by navigating to history and back) does not produce duplicate scan processing — confirming the event listener is correctly removed and re-added

## Notes

- The 50ms timing threshold is empirically derived from the typical operating speed of affordable USB HID barcode scanners commonly sold in Sri Lanka for retail use. If integration testing with a physical scanner shows false positives or false negatives, this value should be tunable via an environment variable or the tenant settings panel added in Phase 05.
- The cache-first approach for barcode lookups is important for performance in high-volume retail environments. If a cashier is scanning the same SKU for 20 units, 19 of those lookups will be satisfied from the TanStack Query cache without any network traffic.
- The hook does not use a hidden input element approach (an alternative implementation pattern) because a hidden input element requires management of focus state, which conflicts with the keyboard navigation patterns of other POS UI elements. The window-level keydown approach with activeElement checking is more robust for this use case.
